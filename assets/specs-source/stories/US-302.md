# US-302 — Indice composito su `tasks(project_id, status)`

## Contesto

L'endpoint `GET /api/v1/tasks` (US-102) filtra quasi sempre su
`project_id` + `status`. Su tenant con migliaia di task in tabella la
query degrada oltre 2 secondi (cfr. BUG-504). Manca un indice mirato.

## Acceptance Criteria

- Creare una migrazione Alembic che aggiunge un indice composito:
  ```sql
  CREATE INDEX CONCURRENTLY idx_tasks_project_status
    ON tasks (project_id, status)
    WHERE archived_at IS NULL;
  ```
- L'indice è **parziale** (escludendo i task archiviati) per ridurre
  dimensione e mantenere le query "hot" veloci.
- Migrazione **reversibile**: `downgrade()` dropa l'indice.
- Commento SQL nella migrazione che riferisce **BUG-504** come motivazione.
- Le query con `WHERE project_id = ? AND status = ?` devono usare l'indice
  (verificare con `EXPLAIN`).
- Tempo di esecuzione migrazione su tabella da 1M righe: < 30s grazie a
  `CREATE INDEX CONCURRENTLY` (no lock prolungato).

## Scenario di accettazione (manuale)

```sql
EXPLAIN ANALYZE
SELECT * FROM tasks
WHERE project_id = '...' AND status = 'open' AND archived_at IS NULL
ORDER BY updated_at DESC
LIMIT 25;
-- Atteso: Index Scan using idx_tasks_project_status
-- Tempo: < 50ms su dataset 1M righe
```

## Note
- ADR-005 richiede commento nella migrazione che spieghi il "perché",
  non solo il "cosa".
